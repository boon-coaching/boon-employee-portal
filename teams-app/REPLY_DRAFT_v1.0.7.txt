Test Blocker #6 — Connect Microsoft Teams loading loop

Root cause: in the OAuth callback, we were saving the
employee_teams_connections row only AFTER a best-effort Bot
Framework "create proactive 1:1 conversation" call. On a fresh
tenant (including FRE) where the bot app is not yet installed on
the user's personal scope, the Bot Framework /v3/conversations
call could stall for long enough that the function redirected
before the connection row was persisted — so the portal showed
"Connect Microsoft Teams" again on return.

Fix in v1.0.7:
  1. employee_teams_connections is now upserted immediately after
     the Graph /me lookup, with an empty conversation_id. The
     portal badge flips to "Connected via Microsoft Teams" on
     redirect regardless of bot install state.
  2. The proactive conversation creation now runs behind a
     7-second AbortController timeout so it cannot stall the
     callback. On success the row is updated with the
     conversation_id.
  3. The conversationUpdate handler backfills conversation_id
     and service_url the moment the bot is added to a personal
     chat, using the user's aadObjectId from the activity.
  4. The portal Settings page now shows a visible success or
     error toast after the redirect and retries the connection
     status query briefly to absorb any read-replica lag.

Net effect: whether the tester (a) connects OAuth before
installing the bot, (b) installs the bot before OAuth, or
(c) connects in a tenant where the app is not yet available,
the portal reflects the connection state correctly within one
redirect.
